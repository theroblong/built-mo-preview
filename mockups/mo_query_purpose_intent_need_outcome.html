Mo by BUILT / Query explainer

Purpose, Intent, Need, and Outcome of Each Query

This page explains what each query or pipeline action contributes to the Mo solution, from raw SPINS normalization through cannibalization scoring, event creation, and price elasticity recommendations.

Cannibalization Price Elasticity Shared Pattern

Cannibalization Pipeline

These steps turn SPINS weekly POS data into comparison pools, donor evidence, scored cannibalization records, and event recommendations.

C0-C17
PrepareNormalize source data and attach stable product, flavor, pack, account, and geography context.
CompareBuild focal/donor/competitor pools so the product can inspect the right substitute set.
ScoreConvert pre/post movement into labels, ML features, confidence, and ranked donor outputs.
ExplainPackage recommendations, event queues, provenance, and ramp guardrails for the UI.
C0

Category Extract Normalization

Output: built_filtered_weekly
Purpose

Normalize raw SPINS fields and narrow to BUILT plus relevant category context.

Intent

Create a stable, reusable base from the raw extract.

Need

Raw column names and mixed retail/geography fields are too brittle for downstream logic.

Outcome

A cleaned weekly source table that preserves direct retail and geography fields.

C1

Enriched Weekly Base Table

Output: built_enriched_weekly
Purpose

Attach product, flavor, pack, brand, and direct retail context.

Intent

Make every weekly row business-readable and model-ready.

Need

Cannibalization depends on accurate product hierarchy and account/geography scope.

Outcome

An enriched weekly panel used by comparison, windowing, scoring, and UI queries.

C2

Universal Comparison Pool

Output: comparison_pool_weekly
Purpose

Build valid focal/candidate SKU pairs for relationship distances 1-5.

Intent

Give UI and ML one governed comparison pool that can be filtered by mode.

Need

Separate pair queries would be slower, inconsistent, and harder to audit.

Outcome

Reusable same-flavor, same-brand, pack-ladder, and adjacent comparison pairs.

C2b

On-Demand Competitive Pool

Output: returned directly to UI
Purpose

Build cross-flavor competitive pairs when the user selects a competitor brand.

Intent

Keep broad competitor analysis available without pre-materializing every pair.

Need

Distance-6 comparisons can become too large if generated for all markets upfront.

Outcome

Tier-sorted competitor comparisons for the selected focal item and market.

C2c

Cross-Flavor Heatmap Aggregation

Output: returned directly to UI
Purpose

Aggregate pre/post change by SKU and geography for cross-flavor diagnosis.

Intent

Show where related flavors are falling, flat, or rising.

Need

Users need to separate product-specific donor loss from broad category softness.

Outcome

A SKU-by-geography heatmap with standardized change buckets.

C2d

Scope Bar Metadata

Output: returned directly to UI
Purpose

Summarize pool type, relationship distance, SKU count, and pair direction.

Intent

Make every diagnosis screen clear about the active comparison set.

Need

Users must know whether they are viewing pack ladder, flavor, brand, or competitor evidence.

Outcome

A compact scope bar that anchors the interpretation of each screen.

C3

Focal Pre/Post Aggregation

Output: built_prepost_features
Purpose

Compute pre/post metrics for each focal SKU across multiple scoring windows.

Intent

Establish whether the focal item gained demand, distribution, velocity, price, or promo support.

Need

Cannibalization cannot be scored from post-period performance alone.

Outcome

Windowed focal evidence for diagnostics, labels, and model features.

C4

Donor Pre/Post Aggregation

Output: donor_prepost_features
Purpose

Apply the same pre/post logic to likely donor SKUs.

Intent

Measure what happened to candidates that may have lost demand.

Need

A focal SKU lift is not cannibalization unless donor movement is visible.

Outcome

Windowed donor evidence paired to focal launch timing.

C5

Final ML Feature Assembly

Output: ml_training_features
Purpose

Join focal and donor features, then compute deltas, labels, confounds, and incremental share.

Intent

Convert evidence into training-ready examples and useful signal columns.

Need

Models need one coherent feature table with deterministic labels and guardrails.

Outcome

Model-ready rows for cannibalization training and score support.

C6

Rolling Stats and Z-Scores

Output: event_detection_weekly
Purpose

Compute rolling averages, standard deviations, z-scores, and week-over-week deltas.

Intent

Detect abnormal demand, velocity, or distribution movement.

Need

Event detection needs trend context, not just static values.

Outcome

Anomaly features for surfacing material events.

C7

New UPC Detection

Output: new_upc_candidates
Purpose

Find BUILT UPCs in the current period that have no prior history.

Intent

Identify new launches automatically.

Need

New products need ramp handling before normal scoring.

Outcome

A candidate list of newly observed UPCs.

C8

New UPC Classification

Output: new_upc_classifications
Purpose

Classify new UPCs as new pack size, new flavor candidate, or relaunch/duplicate.

Intent

Decide how the new item should enter comparison logic.

Need

A new pack size behaves differently from a relaunch or new flavor.

Outcome

Launch classification records for scoring and event routing.

C9

Ramp Monitoring

Output: new_product_ramp_monitor
Purpose

Track first 16 weeks, distribution trend, velocity trend, scoring status, and underperformance.

Intent

Avoid false-positive cannibalization alerts during launch ramp.

Need

Early weeks are dominated by TDP expansion and incomplete baseline history.

Outcome

Ramp status, suppression status, and launch underperformance flags.

C10

Train Cannibalization Classifier

Output: model_cannibal_v3.pkl
Purpose

Train a model to estimate cannibalization status from assembled features.

Intent

Add probabilistic support to deterministic evidence.

Need

Rules alone cannot rank nuanced cases at scale.

Outcome

A classifier that produces cannibalization risk and status support.

C11

Train Donor Ranker

Output: model_ranker_v3.pkl
Purpose

Train a model to rank likely donor SKUs.

Intent

Help users see which SKU is probably losing demand first.

Need

Donor lists need ordering, not just eligibility.

Outcome

Ranked donor candidates for explanation and action screens.

C12

Train Event Detector

Output: model_event_v3.pkl
Purpose

Train a model to identify material events worth surfacing.

Intent

Reduce dashboard hunting by pushing important cases forward.

Need

Users should not scan every SKU/geography combination manually.

Outcome

Event significance support for Mo's priority queue.

C13

Score Production Pairs

Output: scored_cannibalization
Purpose

Generate production scores for focal/account/geography/window pairs.

Intent

Turn model outputs into queryable business records.

Need

UI screens need scored records with confidence, action, and explanation support.

Outcome

Scored cannibalization records with drivers and recommendation fields.

C14

Assemble Event Queue

Output: event_queue
Purpose

Combine scored outputs, suppression rules, and event typing.

Intent

Create the prioritized work queue for Mo.

Need

Recommendations need event-level packaging, not isolated model rows.

Outcome

Business-ready cannibalization events with type, confidence, and suppression logic.

C15

Auto-Enroll New Pack Sizes

Output: pack_ladder_pairs_weekly update
Purpose

Add new pack-size candidates into pack ladder monitoring.

Intent

Keep the system current as the portfolio changes.

Need

New packs should not wait for manual setup before monitoring starts.

Outcome

Updated pack ladder monitoring relationships.

C16

Write New-Pack Events

Output: event_queue
Purpose

Add NEW_PACK_SIZE events to the event queue.

Intent

Surface new pack-size monitoring to users.

Need

Launch events need visible onboarding and scoring status.

Outcome

New-pack events available on the priority landing screen.

C17

Weekly Rescore Trigger

Output: incremental update
Purpose

Refresh incremental scores after each data load.

Intent

Keep Mo current with the latest SPINS feed.

Need

Stale scores can mislead launch, assortment, and pricing decisions.

Outcome

Updated scoring and event records on the weekly cadence.

Price Elasticity Extension

These queries extend the Mo suite with price, promo, pack ladder, competitive gap, benchmark, forecast, and price-defense intelligence.

Q14-Q22
Q14

Price Elasticity Weekly Features

Output: price_elasticity_weekly_features
Purpose

Add price per bar, log price, log units, promo depth bucket, and seasonality controls.

Intent

Build the core weekly feature layer for elasticity modeling.

Need

Raw ARP and units are not enough to model price response cleanly.

Outcome

Weekly price-response features ready for modeling and diagnosis.

Q15

Price Pack Ladder Weekly

Output: price_pack_ladder_weekly
Purpose

Compare price levels between pack sizes of the same specific flavor.

Intent

Show whether the pack ladder has healthy or risky price gaps.

Need

Pack-size pricing can create internal switching or collapse a pack role.

Outcome

Pack ladder price diagnostics by flavor, account, geography, and week.

Q16

Price Competitive Weekly

Output: price_competitive_weekly
Purpose

Compare BUILT price per bar against Tier 1 competitors.

Intent

Separate competitive price pressure from internal BUILT switching.

Need

Users need to know whether softness is caused by competitors or BUILT's own pricing.

Outcome

Competitive price gap diagnostics for selected markets and accounts.

Q17

Price Elasticity Training Features

Output: price_elasticity_training_features
Purpose

Build regression-ready windows for own-price, cross-price, and promo elasticity.

Intent

Prepare clean model inputs with guardrail context.

Need

Elasticity models need usable price movement, adequate weeks, and confound controls.

Outcome

Training set for own-price, cross-price, promo, and forecast models.

Q18

Scored Price Elasticity

Output: scored_price_elasticity
Purpose

Store model outputs, confidence, drivers, and recommended actions.

Intent

Make elasticity results queryable by the UI and Mo explanations.

Need

Users need scored records, not notebook-only model results.

Outcome

Elasticity summary, confidence, drivers, and action labels.

Q19

Price Elasticity Forecast Weekly

Output: price_elasticity_forecast_weekly
Purpose

Store scenario forecasts for price and promo decisions.

Intent

Power the what-if calculator and low/base/high outcomes.

Need

Pricing decisions need forecasted impact before action is taken.

Outcome

Forecast records for planned ARP, promo, and competitor-gap scenarios.

Q20

MULO FOOD Pack Size Norms

Output: mulo_food_pack_size_norms
Purpose

Create MULO FOOD protein bar norms for 1ct, 4pk, 8pk, and 12pk.

Intent

Benchmark BUILT pack pricing and productivity against category norms.

Need

A value pack may look strong internally but still be mispriced against category expectations.

Outcome

Norms for ARP, price per bar, velocity, store selling, and distribution indexes.

Q21

Flavor Protein Driver Features

Output: flavor_protein_driver_features
Purpose

Compare flavor and protein content against sales, velocity, and store penetration.

Intent

Explain whether flavor, protein content, price, or distribution better explains demand.

Need

Protein content can be a driver, but it should not be overclaimed without controls.

Outcome

Driver features for MULO norms, explanation, and pricing context.

Q22

Price Event Queue

Output: price_event_queue
Purpose

Store significant price, promo, new item, benchmark, confidence, and price-defense events.

Intent

Push important pricing situations to the user proactively.

Need

Analysts should not inspect every price chart to find meaningful movement.

Outcome

Prioritized pricing events with trigger, confidence, provenance, drivers, and action.

Prepare

Normalize, enrich, and structure raw source data so every downstream step uses the same definitions.

Compare

Create focal, donor, competitor, geography, and pack-ladder context for business diagnosis.

Score

Convert evidence into features, model outputs, confidence, and recommended actions.

Explain

Surface events, provenance, guardrails, and plain-language reasoning in the UI.

Numbering note: cannibalization is shown as C0-C17 to avoid confusion with the price elasticity extension, which uses Q14-Q22 in the field guide.
