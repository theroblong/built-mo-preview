Part 1a — Pre-Ingest Baseline Check Rob

Before touching spins_full, record the current state. These numbers are your rollback reference.

SQL Baseline check — run before any ingest 
-- 1. Record current high-water mark
SELECT MIN(__time), MAX(__time) FROM "spins_full";
-- Current state: 2023-01-08 → 2026-04-19

-- 2. Row count per year — compare after ingest
SELECT FLOOR(__time TO YEAR) AS yr, COUNT(*) AS rows
FROM "spins_full"
GROUP BY 1
ORDER BY 1;

-- 3. BUILT UPC count — spot new items after ingest
SELECT COUNT(DISTINCT "UPC") FROM "spins_full" WHERE "Brand" LIKE 'BUILT%';

Save these numbers. 2023/2024/2025 row counts must match exactly after ingest. The BUILT UPC count should only grow or stay flat — never shrink.

Part 1b — Druid Ingest: New Weeks Only Rob

Ingest Brian's new file into spins_full using a safe incremental pattern.

✕
Do NOT use OVERWRITE ALL or INSERT INTO

OVERWRITE ALL times out — the cluster's task duration limit is ~1.5h; spins_full at 97M+ rows always fails (E03). INSERT INTO creates duplicate rows if re-run. Only REPLACE INTO … OVERWRITE WHERE is safe.

Step 1 — Confirm the gap boundary

SQL Gap boundary check 
SELECT MAX(__time) AS current_max FROM "spins_full";
-- Example: 2026-04-19T00:00:00.000Z

Brian's new file must start at or before this date. One week of overlap is fine — the OVERWRITE handles the boundary cleanly. If there is a gap (e.g., current max is April 19 but Brian's file starts June 1), stop and resolve with Brian — a gap in spins_full breaks all rolling-window features at that boundary.

Step 2 — Ingest via REPLACE INTO OVERWRITE WHERE

ℹ
Column name mapping (E01)

The CSV has commas in some column names ("Units, Yago"). Druid stores them without the comma ("Units Yago"). Rob's original ingest spec already maps these — use the same spec, just update the MinIO URI and the OVERWRITE WHERE bounds.

SQL Incremental ingest — replace new period only 
-- Replace only the new period. Start 1 week before current MAX to
-- handle the boundary cleanly. Future upper bound must be explicit
-- and DAY-aligned (E20).

REPLACE INTO "spins_full"
OVERWRITE WHERE __time >= TIMESTAMP '2026-04-12'   -- one week before current MAX
            AND __time <  TIMESTAMP '2028-01-01'   -- explicit future bound (E20)
SELECT
  __time,
  "Channel/Outlet"     AS channel_outlet,
  -- ... all column mappings from original ingest spec
FROM EXTERN(
  '{"type":"s3","uris":["s3://mo-ml/spins/All_items_extract_MMDDYY.csv"],...}',
  '{"type":"csv","findColumnsFromHeader":true}'
)
WHERE __time >= TIMESTAMP '2026-04-12'
  AND __time <  TIMESTAMP '2028-01-01'
PARTITIONED BY DAY
CLUSTERED BY "UPC", "Channel/Outlet", "Retail Account", "Geography";

Step 3 — Post-ingest validation (all 6 must pass)

SQL A — New high-water mark 
SELECT MIN(__time), MAX(__time) FROM "spins_full";
-- MAX should reflect the new end date from Brian's export
SQL B — Historical data intact (2023/2024/2025 counts must not change) 
SELECT FLOOR(__time TO YEAR) AS yr, COUNT(*) AS rows
FROM "spins_full"
GROUP BY 1
ORDER BY 1;
-- 2023, 2024, 2025 must match pre-ingest baseline exactly
SQL C — No duplicate weeks in boundary zone 
SELECT __time, COUNT(*) AS row_count
FROM "spins_full"
WHERE __time >= TIMESTAMP '2026-04-01'
GROUP BY __time
ORDER BY __time;
-- Each date should appear exactly as many times as before
-- Double counts = segment not fully replaced; escalate to Rob
SQL D — No gap in weekly cadence 
SELECT __time
FROM "spins_full"
WHERE __time >= TIMESTAMP '2026-01-01'
GROUP BY __time
ORDER BY __time;
-- Dates must be consecutive week-end days with no missing weeks
SQL E — BUILT UPC count (compare to baseline) 
SELECT COUNT(DISTINCT "UPC") FROM "spins_full" WHERE "Brand" LIKE 'BUILT%';
-- Should be ≥ pre-ingest count; unexpected shrink = ingest error
SQL F — Row count in new period is plausible 
SELECT COUNT(*) FROM "spins_full"
WHERE __time > TIMESTAMP '2026-04-19';
-- Should be: (new weeks) × (prior avg weekly row count ± 20%)

Part 2 — New UPC Check Jason

Run before QS1. New BUILT UPCs will appear with NULL flavor, pack count, and size unless QS1 is updated first.

SQL Find BUILT UPCs not yet mapped in flavor_mapping 
SELECT DISTINCT s."UPC", s."Description", s."PACK COUNT", s."FLAVOR"
FROM "spins_full" s
LEFT JOIN "flavor_mapping" fm ON s."UPC" = fm.upc
WHERE s."Brand" IN ('BUILT', 'BUILT BAR', 'BUILT PUFF', 'BUILT SOUR PUFF')
  AND fm.upc IS NULL
ORDER BY s."UPC"

If rows returned: add a WHEN '<new_upc>' THEN ... block in QS1 for each new UPC covering: brand, flavor_family, specific_flavor_raw, specific_flavor_normalized, pack_count, and size. Then re-run QS1.

If 0 rows returned: skip ahead to QS1 re-run — no edits needed.

Q2 — comparison pool (OVERWRITE ALL in annual batches)

Q2 self-joins built_enriched_weekly on flavor/brand to build all focal/candidate pairs. Must be re-run in full because new UPCs enter the pair set.

⚠
Requires special SET commands (E10, E13, E14, E16)

Without these, the self-join fails with OOM or broadcast-limit errors. Annual batches required — never run as a single OVERWRITE ALL.

SQL Q2 — annual batch pattern (run once per year range) 
SET sqlJoinAlgorithm = 'sortMerge';
SET maxNumTasks = 16;
SET rowsPerSegment = 5000000;
-- SET durableShuffleStorage = 'true';  -- uncomment when Rob completes MSQ S3 config (E14)

-- Batch 1: 2023
REPLACE INTO "comparison_pool_weekly"
OVERWRITE WHERE __time >= TIMESTAMP '2023-01-01' AND __time < TIMESTAMP '2024-01-01'
SELECT ...
FROM "built_enriched_weekly" c
JOIN "built_enriched_weekly" f ON ...
WHERE c.__time >= TIMESTAMP '2023-01-01' AND c.__time < TIMESTAMP '2024-01-01'
  AND f.__time >= TIMESTAMP '2023-01-01' AND f.__time < TIMESTAMP '2024-01-01'
  -- ^ both-side time filters required (E17) — sort-merge planner does NOT infer from one side
...
PARTITIONED BY DAY
CLUSTERED BY focal_upc, candidate_upc, retail_account, channel_outlet;

-- Batch 2: 2024-01-01 → 2025-01-01
-- Batch 3: 2025-01-01 → 2026-01-01
-- Batch 4: 2026-01-01 → 2028-01-01  (adjust upper bound annually)

Known Constraints & Risks

RiskDetailMitigation
OVERWRITE ALL timeout Cluster task limit ~1.5h; spins_full at 97M+ rows always fails (E03) Use REPLACE INTO … OVERWRITE WHERE with year-range batches — never OVERWRITE ALL for spins_full
INSERT INTO duplicates Not idempotent — re-running doubles every row in the period Use REPLACE INTO … OVERWRITE WHERE for all spins_full updates
Data gap in Brian's file Export start doesn't connect to current MAX(__time) Run gap boundary check before ingest; stop and resolve if gap found
Column names (E01) CSV: "Units, Yago" → Druid: "Units Yago" (13 columns affected) Rob's original ingest spec already maps these; use the same spec
Wrong extract format 167-column file missing 18 Q0 required fields Always use 214-column format; verify count before depositing
New BUILT UPCs Appear as NULL in flavor_mapping if QS1 not updated Run new-UPC check (Part 2) before QS1
New competitor brands New brand won't have a tier in item_catalog Compare SELECT DISTINCT source_brand FROM built_filtered_weekly after Q0 against current QS3 list
Q2 sort-merge OOM (E13–E14) Self-join on 62M+ rows; worker pods OOM-evicted without durable shuffle SET sqlJoinAlgorithm='sortMerge'; SET maxNumTasks=16; + annual batches; Rob to complete MSQ S3 shuffle config (E14)
Q2 OVERWRITE WHERE alignment (E20) Open-ended upper bound → INVALID_INPUT Always use explicit future bound: AND __time < TIMESTAMP '2028-01-01'
Q2 both-side filter (E17) Time filter on one join side not inferred by sort-merge planner Add explicit c.__time AND f.__time bounds in WHERE
appendToExisting: false Silently drops columns → UI returns empty arrays All MO_xx write-backs use appendToExisting: true; never override

v4 retrain trigger No new SPINS data → retraining produces same rows, collapsed holdout Only retrain when new data extends the date range in spins_full
